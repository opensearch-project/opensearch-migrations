# Generated K8s Resource Handler Contract

## Goal

Replace the current mix of:
- bulk `kubectl create -f crdResources.yaml`
- separate gate cleanup/create scripts
- batch status patch scripts

with one generated script that handles every generated K8s resource individually and correctly.

Suggested output name:

- `handleK8sResources.sh`

This script should be generated by the initializer and invoked by `createMigrationWorkflowFromUserConfiguration.sh`.

## High-level behavior

The script should process resources one at a time and emit a clear result for each resource.

For every generated resource, it should:

1. identify the resource by:
   - kind
   - apiVersion
   - namespace
   - name
2. decide which lifecycle rule applies:
   - approval gate
   - root migration CR
   - config map / helper resource
3. perform the correct operation for that resource type
4. emit an explicit result such as:
   - `CREATED`
   - `UPDATED`
   - `UNCHANGED`
   - `ALREADY_EXISTS`
   - `FAILED`
   - later, possibly:
     - `GATED`
     - `IMPOSSIBLE`
5. fail fast on real errors instead of swallowing stderr

## Resource categories

### 1. ApprovalGate resources

Approval gates should be treated as ephemeral control resources.

Per gate:
1. delete stale gate if it exists
2. create gate
3. patch initial status
4. emit:
   - `CREATED ApprovalGate <name>`

This preserves the current model:
- gates are always recreated
- stale approval state does not survive submit
- gate lifecycle is independent of the main CR graph

### 2. Root migration CRs

These are the durable contract resources:
- `KafkaCluster`
- `CapturedTraffic`
- `DataSnapshot`
- `SnapshotMigration`
- `TrafficReplay`

Near-term behavior:
- handle each CR individually
- do not bulk-create them
- patch status individually only when create succeeds

Full target behavior:
- perform CR-first reconcile
- distinguish:
  - created
  - unchanged / identical
  - updated
  - gated by VAP
  - impossible change
  - failed

This is the category that should evolve into the full CR-first reconciliation model.

### 3. ConfigMaps / helper resources

Examples:
- approval config maps
- concurrency config maps

These can usually stay on:
- `kubectl apply`

But still should be handled individually or by a clearly scoped batch with real stderr preserved.

## Near-term implementation contract

To keep the first version tractable, the generated script can use this behavior:

### ApprovalGate
- always delete and recreate
- always patch initial status

### Root CR

For each CR:
1. attempt `kubectl create -f <resource>`
2. if create succeeds:
   - patch status for that resource
   - emit `CREATED`
3. if create fails because the resource already exists:
   - emit `ALREADY_EXISTS`
   - do not patch status
4. if create fails for any other reason:
   - emit `FAILED`
   - print stderr
   - exit nonzero

Important:
- detection of `already exists` must be explicit
- do not treat all failures as `already exists`

### ConfigMap / helper resource
- `kubectl apply -f <resource>`
- emit `CREATED` or `UPDATED` / `UNCHANGED` if detectable
- fail on real errors

This near-term version already fixes the current sharp edges:
- no bulk partial-create ambiguity
- no swallowed stderr
- no whole-batch status patch skip

## Full target contract for root CRs

Once CR-first reconciliation is added, root CR handling should evolve to:

For each root CR:
1. load desired manifest
2. inspect existing resource
3. compare desired vs existing for VAP-relevant spec fields
4. attempt reconcile
5. emit one of:
   - `CREATED`
   - `UNCHANGED`
   - `UPDATED`
   - `GATED`
   - `IMPOSSIBLE`
   - `FAILED`

Semantics:
- `CREATED`
  - new CR created successfully
- `UNCHANGED`
  - resource already exists and desired spec is effectively identical
- `UPDATED`
  - safe or approved update applied successfully
- `GATED`
  - update blocked by VAP and requires approval/retry
- `IMPOSSIBLE`
  - attempted forbidden immutable change
- `FAILED`
  - unexpected operational failure

This is the result vocabulary the workflow/submit layer should eventually key off of.

## Output format

Keep the output machine-readable enough to debug and extend later.

Recommended line format:

```sh
RESULT <kind> <name> <state>
```

Examples:
- `RESULT KafkaCluster source-kafka CREATED`
- `RESULT SnapshotMigration source.target.snap1.migration-0 ALREADY_EXISTS`
- `RESULT ApprovalGate source.target.snap1.migration-0.evaluateMetadata CREATED`
- `RESULT TrafficReplay source-target-replay FAILED`

For failures, also print stderr and a summary line like:
- `ERROR TrafficReplay source-target-replay validation failed`

If you want a richer format later, emit JSON lines instead.

## File generation shape

I do not think this needs a strong decision up front.

The important requirement is:
- one generated handler script
- per-resource handling inside that script
- explicit resource metadata available to drive the per-resource logic

Either of these shapes is acceptable:

### Option A: per-resource files

Example temp dir contents:
- `resources/000-kafkacluster-source.yaml`
- `resources/001-capturedtraffic-source.yaml`
- `resources/002-approvalgate-foo.yaml`
- `status/000-kafkacluster-source-status.yaml`
- `status/002-approvalgate-foo-status.yaml`
- `handleK8sResources.sh`

Then the script loops over these explicitly.

### Option B: one manifest index plus one script

Example:
- `resourceIndex.json`
- `handleK8sResources.sh`

Where `resourceIndex.json` lists:
- file path
- kind
- name
- category
- whether status patch is needed

Then the script reads that index and applies the correct per-resource logic.

Choose whichever shape is simpler to implement in the initializer.
The design does not depend on a hard preference here.

## Invocation contract

`createMigrationWorkflowFromUserConfiguration.sh` should become much simpler:

1. run initializer
2. invoke:
   - `handleK8sResources.sh`
3. invoke:
   - `enrichWorkflowConfigWithUids.sh`
4. create the workflow

That replaces the current scattered handling of:
- `cleanupApprovalGates.sh`
- `crdResources.yaml`
- `patchCrdStatus.sh`
- `approvalGates.yaml`
- `patchApprovalGateStatus.sh`

## Error-handling rules

The generated script should:

- never discard stderr from `kubectl`
- only special-case errors it can positively identify
- exit nonzero on unexpected failures
- not print `already exists` unless that specific condition was observed

For create operations, that means:
- inspect the actual `kubectl` error text / return path
- only map exact `AlreadyExists` cases to `ALREADY_EXISTS`

## Why this helps

This design fixes the current problems:

- no more bulk CR create with partial success ambiguity
- no more whole-batch skip of status patching
- no more swallowing validation/RBAC errors
- one place to evolve from create-only behavior into CR-first reconcile behavior
- a consistent control point for both ApprovalGates and root CRs

## Recommended implementation order

1. generate `handleK8sResources.sh`
2. move gate cleanup/create/status patch into that script
3. move root CR create/status patch into per-resource handling in that script
4. simplify `createMigrationWorkflowFromUserConfiguration.sh` to call only the generated handler
5. later evolve root CR handling from create-only into full reconcile semantics
