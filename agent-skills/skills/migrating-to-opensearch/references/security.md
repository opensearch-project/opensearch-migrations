# Security Considerations

Every assessment report MUST include a Security section covering all of the following recommendations. Verify against the Amazon OpenSearch Service security best practices page at runtime — retrieve via [`knowledge-retrieval.md`](knowledge-retrieval.md) (Amazon OpenSearch Service (managed) section).

## Contents

- Auth — IAM SigV4, short-lived credentials, scoped actions
- Authorization — FGAC; never anonymous
- Transport — TLS 1.2+, ACM custom-endpoint cert
- Web-facing security headers — HSTS, X-Frame-Options, CSP
- Encryption at rest — customer-managed KMS, node-to-node, SSE-KMS S3
- Network — VPC, no `0.0.0.0/0`, AWS WAF, VPC endpoints
- Audit — CloudTrail, audit logs, SNS+KMS for alarms
- Migration-tool security — least-privilege EKS/ECS roles, Secrets Manager
- Secrets — never hardcode
- Sensitive data in reports
- Production-like defaults
- Input validation, throttling, rate limiting (3-layer defense)

- **Auth.** IAM with SigV4 signing as the default (managed: service `es`; Serverless: service `aoss`). You MUST recommend IAM roles delivering short-lived credentials over IAM users with long-lived access keys because long-lived keys are the most common credential-leak vector. Use specific IAM actions (e.g. `es:DescribeDomain`, `es:ESHttpGet`, `es:ESHttpPost`, `aoss:APIAccessAll`, `aoss:CreateIndex`, `aoss:WriteDocument`, `osis:CreatePipeline`); you MUST NOT recommend `es:*`, `aoss:*`, `osis:*`, or `*FullAccess` managed policies in production because they grant write/admin permissions that bypass least-privilege controls. Include condition keys (`aws:SourceArn`, `aws:SourceAccount`) where applicable. For cross-account snapshot S3 access, scope the bucket policy to the migration role's principal.
- **Authorization.** Fine-grained access control (FGAC) with IAM principals, SAML, or OIDC. You MUST NOT recommend anonymous access because it removes all per-user audit trail. For the assessment phase the customer needs only read-only describe permissions; you SHOULD reject any guidance that grants write access during assessment.
- **Transport.** Set `EnforceHTTPS: true` and select the most current `TLSSecurityPolicy` value published in the Amazon OpenSearch Service developer guide — verify the live policy name against the doc per [`knowledge-retrieval.md`](knowledge-retrieval.md) (Amazon OpenSearch Service (managed) section). The data-protection page requires TLS 1.2 minimum and recommends TLS 1.3. For migration-tooling endpoints (Capture Proxy, OSI pipelines), you MUST require TLS termination on the AWS side and certificate validation on the source. For custom domain endpoints, you SHOULD provision an ACM-issued certificate; the OpenSearch Service custom-endpoint doc explicitly supports ACM-issued certificates. (Verify any rotation guarantee against the live custom-endpoint doc — see [`knowledge-retrieval.md`](knowledge-retrieval.md) (Amazon OpenSearch Service (managed) section).) You MUST NOT recommend self-managed certificates without an automated rotation pipeline because manually-rotated certificates can expire silently and break TLS in production.
- **Web-facing security headers.** When OpenSearch Dashboards or a search API is exposed via ALB, CloudFront, or API Gateway, you MUST recommend a response-headers policy that sets HSTS (`Strict-Transport-Security: max-age=31536000; includeSubDomains`), `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, and a restrictive `Content-Security-Policy`. You MUST NOT skip these headers because Dashboards is a browser-rendered UI and missing security headers expose customers to clickjacking, MIME-type confusion, and downgrade attacks.
- **Encryption at rest.** Enable encryption at rest on the OpenSearch domain (`EncryptionAtRestOptions.KmsKeyId`); you SHOULD use a customer-managed KMS key over the AWS-owned key for separation of duty and audit visibility. Per the AWS encryption-at-rest developer guide page, this protects all indexes (including UltraWarm), OpenSearch logs, swap files, and **automated** snapshots — manual snapshots are NOT covered by the domain key, so the snapshot S3 bucket MUST be independently encrypted (SSE-KMS preferred; SSE-S3 minimum) with public-access blocks enabled. Enable node-to-node encryption (`NodeToNodeEncryptionOptions.Enabled: true`). CloudWatch Log Groups for slow logs and audit logs MUST be KMS-encrypted.
- **Network.** VPC deployment over public-internet whenever the workload allows. You MUST NOT recommend `0.0.0.0/0` in security groups or domain access policies because it exposes the cluster to the entire internet. For non-VPC (public) domains, place AWS WAF in front via CloudFront or ALB to rate-limit and filter malicious requests. For Serverless collections, use VPC endpoints (`aoss:CreateVpcEndpoint`) when private access is required.
- **Audit.** CloudTrail for control-plane API calls. OpenSearch audit logs are available **only when fine-grained access control is enabled** and are published to CloudWatch Logs (per the AWS audit-logs developer guide page); slow-query / application logs are independent and also publish to CloudWatch Logs. Slow-query logs may contain PII from query parameters; you MUST recommend reviewing log content, considering redaction, and restricting log-group access (`logs:GetLogEvents` scoped by IAM). SNS topics used for CloudWatch Alarm notifications MUST be KMS-encrypted (`KmsMasterKeyId`). You MUST scope SNS topic policies (`sns:Subscribe`, `sns:Publish`) to authorized principals only because alarm notifications carry cluster names, index names, and error details that are sensitive cluster metadata. Verify against the live audit-logs doc per [`knowledge-retrieval.md`](knowledge-retrieval.md) (Amazon OpenSearch Service (managed) section).
- **Migration-tool security.** Migration Assistant deploys to EKS or ECS — follow least-privilege task/pod role design. You MUST NOT reuse the cluster admin role for Migration Assistant tasks because that effectively grants the tool full cluster control with no separation of duty. Capture Proxy keys MUST be stored in Secrets Manager, never on disk in plaintext. The traffic-replayer S3 bucket MUST be SSE-KMS-encrypted with the same KMS key as the target domain.
- **Secrets.** You MUST NOT hardcode credentials, endpoints, or master usernames in examples or generated reports because reports are commonly shared with stakeholders and may end up in source control or chat history. Route via AWS Secrets Manager or Parameter Store SecureString. Verify any user-supplied artifacts (e.g. `elasticsearch.yml`) are scrubbed before paste.
- **Sensitive data in the assessment report.** The fingerprint JSON may include cluster names, index names, sample documents, and slow-query examples. Treat the rendered report as customer-confidential. You MUST NOT commit reports to public repos because the contents may include PII or proprietary data.
- **Production-like defaults.** You MUST use plausible demo values (e.g. `r6g.2xlarge.search`, `us-east-1`) and you MUST NOT use `production-`-prefixed names, `1234567890` placeholder IDs, or overly permissive defaults in any example because customers tend to copy examples verbatim into runbooks.
- **Input validation, throttling, and rate limiting (defense in depth).** You MUST recommend all three layers — domain, application, and edge — so a misbehaving client or attacker cannot exhaust the cluster:
  1. **OpenSearch Service request throttling.** Surface HTTP 429 responses to the client by enabling cluster-level circuit breakers (`indices.breaker.*`), search/bulk thread-pool queue caps, and search backpressure. For Serverless, set per-collection ingestion / search OCU caps to bound runaway cost.
  2. **Application-level rate limiting.** Have the customer's application enforce per-tenant or per-API-key rate limits before hitting the OpenSearch endpoint, especially for free-form `query_string` / `simple_query_string` traffic. Reject overly broad queries (`*:*` against large indices, deep `from`+`size` pagination) at the application boundary.
  3. **AWS API Gateway throttling for client-facing search APIs.** When OpenSearch is exposed through a customer-facing API, place AWS API Gateway in front and enable per-stage throttling (`burstLimit` / `rateLimit`) plus AWS WAF rate-based rules.
